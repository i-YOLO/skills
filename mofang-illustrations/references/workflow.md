# 工作流

## 1. 消化内容

提炼核心观点、认知转折、值得配图的段落和只适合文字表达的部分。不要平均配图；优先选择系统边界、信号与噪音、杠杆、反馈、多 agent、从混乱到结构和长期主义判断。

## 2. 选择执行路线

- 只要策略：输出 shot list 或 brief，停止在文本阶段。
- 要真实图片：建立 shot list 和全部 prompts，再选择执行策略。
- 要修改现有图片：检查编辑目标，锁定不变项后调用内置 `image_gen` 编辑。

### 执行策略

- `balanced`：默认。3–8 张时先生成并检查 1 张风格锚点，通过后将剩余资产放入最大并发数 3 的首轮批次。
- `fast`：用户明确要求尽快、快速或并发时使用。所有互不依赖的首轮资产直接进入最大并发数 3 的批次。
- `strict`：1–2 张、首次建立角色、复杂文字、世界观设定或高一致性任务使用。逐张生成并检查。

同一张图片的连续编辑始终串行。不同图片互不依赖时才允许并发。

## 3. 建立 shot list

常规内容输出 4–8 张，短内容输出 1–3 张。每张包含：

- 放置位置。
- 核心意思。
- 构图模式。
- 墨方的核心动作。
- 保留项、可改项和禁止项。
- 画幅比例。
- 是否需要真实生成。

为每张图分配稳定的 `asset_id` 和目标文件名。`asset_id` 在 shot list、prompt、运行清单、迭代稿和最终文件之间保持一致。

## 4. 准备生成

1. 将社媒头像作为角色身份参考。
2. 按 prompt 模板一次性整理全部单图需求。
3. 默认无文字、16:9、浅底手绘风。
4. 在首次调用前创建 `brief/run-manifest.md`，写入执行策略、并发数、开始时间和全部资产的计划状态。
5. 确认每个调用只包含一个资产的 prompt、参考图角色和稳定 `asset_id`。
6. 不因“批量”切换 CLI，不使用外部后端或自定义 API 脚本。

## 5. 执行首轮批次

### balanced

1. 选择最能代表角色动作与正文风格、且不含复杂文字的资产作为锚点。
2. 单独生成锚点并执行完整 QA。
3. 锚点通过后，以最大并发数 3 调用剩余资产；一张图片保持一次独立 `image_gen` 调用。
4. 锚点失败时只修正锚点，不启动剩余批次；通过后再继续。

### fast

1. 将全部互不依赖的首轮资产放入同一任务池。
2. 以最大并发数 3 执行；超过 3 张时分波次补位，任一任务完成后再启动下一项。
3. 不等待单张 QA 后才启动下一张。

### strict

逐张调用、逐张 QA；当前资产通过或完成一次定向处理后再生成下一张。

每个调用结束后按 `asset_id` 写回原始输出路径和状态。单张失败不得取消仍在运行或尚待执行的其他资产。

## 6. 失败恢复

1. 等当前批次结束后收集超时、限流和工具失败资产。
2. 保持原 prompt 和参考图不变，只重试失败资产一次。
3. 将重试批次的最大并发数降为 2。
4. 再次失败时标记为 `failed`，记录错误摘要，不阻塞其他成功资产的 QA 和交付。

## 7. 批量评图与迭代

1. 等首轮批次全部完成后，按 QA Checklist 统一检查角色、动作、结构、原创性、比例、留白、色彩和文字。
2. 将结果标记为 `pass`、`rework`、`rejected` 或 `failed`。
3. 对 `rework` 资产只选择一个主要问题，例如“墨方更主动”“减少节点”“移除标题”或“换掉旧案例空间关系”。
4. 不同资产的单变量返修可以组成最大并发数 3 的返修批次；同一资产的多轮返修必须串行。
5. 每轮结束后重新执行完整 QA，并更新迭代次数、状态和最终路径。

## 8. 交付

- 预览：内联展示即可。
- 正式交付：保存最终图、版本化迭代图、shot list、最终 prompt 和运行清单。
- 完成后在运行清单填写结束时间和真实墙钟耗时；不要用各图片耗时之和代表并发任务总耗时。
- 回复用户时先给成品与绝对路径，再简述生成策略、失败恢复和筛选结果。
