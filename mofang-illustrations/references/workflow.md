# 工作流

## 1. 消化内容

提炼核心观点、认知转折、值得配图的段落和只适合文字表达的部分。不要平均配图；优先选择系统边界、信号与噪音、杠杆、反馈、多 agent、从混乱到结构、长期主义判断，以及需要解释的流程、机制、数据或教育概念。

长内容只选择 1-6 个认知锚点。不要把每一段都变成一张图。

## 2. 选择执行路线

- 用户明确只要策略、shot list、brief 或 prompts：输出文本方案，停止在文本阶段。
- 用户说“做配图”“生成配图”“正文配图”“解释图”“图表美化”“批量配图”“做 N 张图”等：默认视为真实生图意图，先发出完整 preflight 确认面板。
- 用户说“测试流程”：先发出完整 preflight 确认面板，用来测试确认流程；不要直接输出 shot list。
- 要真实图片：完成 preflight 确认后，再建立 shot list、prompts 和运行清单，并调用 `image_gen`。
- 要修改现有图片：检查编辑目标，锁定不变项后调用内置 `image_gen` 编辑。

## 3. 生图前确认面板

真实调用 `image_gen` 前，必须一次性发出完整 preflight 确认面板。不要一个问题一个问题地拆开问。

每个问题提供推荐项。用户回复“默认”或“推荐”时，采用全部推荐项。用户只回答部分选项时，未回答项采用推荐值，但正式落盘目录和自定义标签除外。

如果上下文已经明确某项选择，可以跳过该问题，但在生图前用一句话说明采用的默认值。

完成必要 preflight 之前，不要创建输出目录、写入 brief、写 prompts、生成 run manifest、调用 `image_gen` 或安排子 Agent 生图。

### 完整确认面板

```markdown
生图前确认：

1. 视觉风格【单选】
- 让 Agent 判断（推荐，Agent 会说明选择理由）
- 手绘系统草图风（适合个人 IP、系统思维、AI 协作、长期主义、正文配图）
- 3D 材质解释图风（适合流程图、机制图、图表美化、教育解释、带标签中心图）

2. 生成节奏【单选】
- 先测试 1 张（推荐）
- 不测试，直接生成全部图片
- 暂不生图，只输出 shot list / prompts

3. 多图执行【单选，仅多图任务有效】
- 平衡模式（推荐，适合 3-8 张：先测试 1 张，通过后并发生成其余图）
- 快速模式（适合 8 张以上或快速探索：多路并行生成，主 Agent 最后统一收口）
- 稳定模式（适合 1-2 张或重要成品：逐张生成和检查）

4. 图内文字【单选】
- Agent 先给推荐标签，你确认后再生成（推荐用于解释图/流程图/机制图）
- 无文字，只用画面表达
- 图内不加文字，文字放到外层卡片/文章/PPT
- 我提供或修改标签

5. 交付形式与保存位置【单选】
- 仅预览（推荐）
- 正式落盘到 Agent 推荐目录（Agent 先给出具体路径，等待你确认后才创建）
- 正式落盘到我指定目录（请直接回复目录路径）
- 只要 prompt，不调用 Image Gen
```

单图任务忽略第 3 项，默认稳定模式。

选择后映射到 `references/style-registry.md` 中的 `style_id`。

用户选择“只要 prompt”时，停止在策略模式，不调用 `image_gen`。

用户选择“暂不生图”时，停止在策略模式，不调用 `image_gen`。

选择“仅预览”或“只要 prompt”都不得创建输出目录。

如果用户选择“正式落盘到 Agent 推荐目录”，Agent 先给出具体推荐路径并等待用户确认；确认前不要创建目录。

如果用户选择“正式落盘到我指定目录”但没有提供路径，不调用生图，继续追问目录路径。

用户确认推荐目录或提供自定义目录后，才读取 `references/asset-delivery.md` 并创建 `brief/`、`iterations/` 或 `final/`。

解释图、流程图、机制图、图表美化和教育解释场景，如果用户选择“Agent 先给推荐标签”，先输出 3-5 个短标签候选并等待确认。用户确认后才把逐字标签写入 prompt。

如果用户选择“我提供或修改标签”但没有给出标签，不调用生图，继续追问标签。

内部执行映射：

- 平衡模式 -> `balanced`
- 快速模式 -> `fast`
- 稳定模式 -> `strict`

## 4. 建立 shot list

常规内容输出 4-8 张，短内容输出 1-3 张。每张包含：

- 放置位置。
- 核心意思。
- 风格 ID。
- 构图模式。
- 核心动作或解释结构。
- 是否需要图内文字和候选标签。
- 保留项、可改项和禁止项。
- 画幅比例。
- 是否需要真实生成。

为每张图分配稳定的 `asset_id` 和目标文件名。`asset_id` 在 shot list、prompt、运行清单、迭代稿和最终文件之间保持一致。

## 5. 准备生成

1. 根据 style registry 选择 prompt 模板和参考图策略。
2. 手绘系统草图风使用社媒头像作为角色身份参考；3D 材质解释图风默认不使用头像参考。
3. 按 prompt 模板统一整理全部单图需求。
4. 在首次调用前创建 `brief/run-manifest.md`，写入风格、执行策略、并发数、开始时间和全部资产的计划状态。
5. 确认每个调用只包含一个资产的 prompt、参考图角色和稳定 `asset_id`。
6. 不因“批量”切换 CLI，不使用外部后端或自定义 API 脚本。

## 6. 执行首轮批次

### balanced

1. 选择最能代表角色动作、解释结构或视觉风格，且不含复杂文字的资产作为测试/锚点图。
2. 单独生成锚点并执行完整 QA。
3. 锚点通过后，以最大并发数 3 调用剩余资产；一张图片保持一次独立 `image_gen` 调用。
4. 锚点失败时只修正锚点，不启动剩余批次；通过后再继续。

### fast

1. 用于 8 张以上、多方向探索或用户明确选择快速模式。
2. 可将互不依赖的资产拆给多个子 Agent。主 Agent 先制定统一 brief、风格规则、asset_id、标签、参考线索和 QA 标准。
3. 每个子 Agent 只负责自己的资产生成和初筛，不决定最终交付。
4. 主 Agent 收回所有结果后统一评图、返修、命名和交付。
5. 如果当前环境没有可用子 Agent，则退回最大并发数 3 的主 Agent 并发。

### strict

逐张调用、逐张 QA；当前资产通过或完成一次定向处理后再生成下一张。适合 1-2 张、首次建立角色、复杂文字、世界观设定、高一致性或重要成品。

每个调用结束后按 `asset_id` 写回原始输出路径和状态。单张失败不得取消仍在运行或尚待执行的其他资产。

## 7. 失败恢复

1. 等当前批次结束后收集超时、限流和工具失败资产。
2. 保持原 prompt 和参考图不变，只重试失败资产一次。
3. 将重试批次的最大并发数降为 2。
4. 再次失败时标记为 `failed`，记录错误摘要，不阻塞其他成功资产的 QA 和交付。

## 8. 批量评图与迭代

1. 等首轮批次全部完成后，按 QA Checklist 统一检查风格、身份、动作、结构、原创性、比例、留白、色彩、文字、数据和参考事实。
2. 将结果标记为 `pass`、`rework`、`rejected` 或 `failed`。
3. 对 `rework` 资产只选择一个主要问题，例如“核心动作更明确”“减少节点”“移除标题”“修正标签”“修正数值”或“换掉旧案例空间关系”。
4. 不同资产的单变量返修可以组成最大并发数 3 的返修批次；同一资产的多轮返修必须串行。
5. 每轮结束后重新执行完整 QA，并更新迭代次数、状态和最终路径。

## 9. 交付

- 预览：内联展示即可。
- 正式交付：保存最终图、版本化迭代图、shot list、最终 prompt 和运行清单。
- 完成后在运行清单填写结束时间和真实墙钟耗时；不要用各图片耗时之和代表并发任务总耗时。
- 回复用户时先给成品与绝对路径，再简述风格、生成策略、失败恢复和筛选结果。
